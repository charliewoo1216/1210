전체 설계 요약
A · B · C · D 핵심 결론

A. Kafka 토픽/파티션

  • mes.raw 12 파티션 (6,000 msg/s 기준)
  • 전 토픽 RF=3 (내구성 보장)
  • Producer: acks=all + idempotence
  • Consumer: 수동 commit + earliest reset

B. Parser 플러그인

  • BaseParser 추상 인터페이스 통일
  • subprocess 격리로 플러그인 오류 차단
  • 파일 mtime 기반 핫로딩 (재시작 불필요)
  • timeout=10s 강제 종료 보호

C. LLM Selector

  • qwen3:8b + Ollama, format=json 강제
  • confidence ≥ 0.85 → 직행
  • 0.60~0.85 → fallback parser + 로그
  • unknown / <0.60 → retry or dlq

D. Retry / DLQ

  • 최대 3회, Backoff 0s/30s/5m
  • failed_stage 기반 최적 재투입 토픽
  • DLQ 100건 초과 시 Slack 알림
  • 30일 보존 → 수동 재처리 여유
